## 1. Architecture Goals

### 1.1 Primary Goals (P0)

1. **End-to-end deploy chain**: `infracast deploy` from user code to running service on ACK Serverless (Alibaba Cloud)
2. **Zero IaC for developers**: Platform handles all infrastructure provisioning via SDK Direct
3. **Idempotent state management**: Safe to retry any operation without side effects
4. **Encore compatibility**: Generate valid `infracfg.json` that Encore runtime consumes without modification

### 1.2 Architecture Constraints (Frozen)

These constraints derive from PRD v1.1 frozen decisions and are non-negotiable:

| # | Constraint | Source |
|---|-----------|--------|
| C1 | Compute target = ACK Serverless only | PRD §1, Decision 1 |
| C2 | `infracast.yaml` with required `provider` + `region` | PRD §1, Decision 2 |
| C3 | `infracfg.json` generated by CLI, aligned with Encore schema | PRD §1, Decision 3 |
| C4 | SDK Direct for resource provisioning (no Terraform in P0) | PRD §1, Decision 5 |
| C5 | No Encore IR dependency in P0 | PRD §1, Decision 6 |
| C6 | Redis Stream as Pub/Sub fallback (no native RocketMQ/TDMQ in P0) | PRD §1, Decision 7 |
| C7 | Forward-only DB migration | PRD §1, Decision 9 |
| C8 | No Encore Runtime fork in P0 | PRD §1, Decision 10 |
| C9 | STS/AssumeRole for cloud credentials | PRD §7.2 |
| C10 | `(env_id, resource_name)` + `spec_hash` + `state_version` idempotency | PRD §7.2 |

### 1.3 Quality Attributes

| Attribute | Target | Rationale |
|-----------|--------|-----------|
| **Extensibility** | Add new cloud provider without modifying core | Provider Registry pattern |
| **Idempotency** | All provisioning operations are safe to retry | spec_hash + state_version |
| **Observability** | Every deploy step emits structured logs | Debugging failed deployments |
| **Testability** | Mock provider enables full pipeline testing without cloud calls | CI speed + cost |
| **Simplicity** | Minimal abstraction layers in P0 | Ship fast, refactor later |

### 4.2 Build Metadata Schema

The `BuildMeta` struct captures what `encore build` outputs. This is the contract between the build step and the Service Mapper (replacing Encore IR dependency per constraint C5).

```go
// internal/mapper/buildmeta.go
type BuildMeta struct {
    AppName      string         `json:"app_name"`
    Services     []ServiceMeta  `json:"services"`
    Databases    []DatabaseMeta `json:"databases"`
    Caches       []CacheMeta    `json:"caches"`
    ObjectStores []ObjectStoreMeta `json:"object_stores"`
    PubSubTopics []PubSubMeta   `json:"pubsub_topics"`
    BuildCommit  string         `json:"build_commit"`
    BuildImage   string         `json:"build_image"`
}

type ServiceMeta struct {
    Name     string `json:"name"`
    EntryPkg string `json:"entry_pkg"` // Go package path
    Port     int    `json:"port"`
}

type DatabaseMeta struct {
    Name          string `json:"name"`
    MigrationDir  string `json:"migration_dir"`
}

type CacheMeta struct {
    Name        string `json:"name"`
    ClusterName string `json:"cluster_name"`
}

type ObjectStoreMeta struct {
    BucketName string `json:"bucket_name"`
}

type PubSubMeta struct {
    TopicName   string   `json:"topic_name"`
    Subscribers []string `json:"subscribers"`
}
```

**How BuildMeta is populated (P0)**:
1. Parse `infracast.yaml` for declared resources
2. Scan Go source for Encore resource declarations (`sqldb.NewDatabase`, `cache.NewCluster`, `objects.NewBucket`)
3. Merge: yaml declarations take precedence over source scan results

**P1 enhancement**: Parse Encore IR (if/when schema stabilizes) for richer metadata.

## 5. Key Architectural Decisions

### ADR-1: CLI-first, no server in P0

**Decision**: Infracast P0 is a CLI tool that runs locally. No API server, no web portal.

**Why**: Reduces scope dramatically. Developer runs `infracast deploy`, CLI calls cloud APIs directly. This is how Encore's self-hosted mode works — the reference architecture we're following.

**Consequence**: State is stored locally (SQLite) or in a shared PostgreSQL instance. Multi-user coordination deferred to P3.

### ADR-2: SQLite for local state, PostgreSQL for team state

**Decision**: Default state backend is SQLite (file: `.infracast/state.db`). Team/shared environments use PostgreSQL.

```go
// internal/state/store.go
type Store interface {
    GetResource(envID, resourceName string) (*ResourceRecord, error)
    UpsertResource(record *ResourceRecord) error  // spec_hash + state_version check
    ListResources(envID string) ([]ResourceRecord, error)
    DeleteResource(envID, resourceName string) error
}

type ResourceRecord struct {
    EnvID              string
    ResourceType       string    // database | cache | object_storage | compute
    ResourceName       string    // unique within env
    ProviderResourceID string    // cloud vendor's resource ID
    SpecHash           string    // SHA-256 of spec fields only
    StateVersion       int       // optimistic locking
    ConfigJSON         []byte    // resource config snapshot
    Status             string    // pending | provisioned | failed | destroyed
    CreatedAt          time.Time
    UpdatedAt          time.Time
}
```

**Why**: SQLite makes the tool self-contained for solo developers. No external DB needed to start. Team mode uses PostgreSQL (same as PRD data model).

**Consequence**: State Store interface abstracts the backend. Two implementations: `sqlite.Store` and `postgres.Store`.

### ADR-3: Provider Registry with compile-time registration

**Decision**: Providers register at program startup via `init()` functions.

```go
// providers/alicloud/register.go
func init() {
    providers.DefaultRegistry.Register(&AlicloudProvider{})
}

// cmd/infracast/main.go
import (
    _ "github.com/DaviRain-Su/infracast/providers/alicloud"  // register
    _ "github.com/DaviRain-Su/infracast/providers/mock"       // register
)
```

**Why**: Simple, Go-idiomatic. No plugin system complexity in P0. Adding a new provider = new package + import in main.go.

**Consequence**: All providers are compiled into the binary. Dynamic loading deferred to P2+ if needed.

### ADR-4: Provisioner idempotency via spec_hash

**Decision**: Before provisioning, compute `spec_hash` of the resource spec (excluding metadata/timestamps), compare with stored hash.

```
hash_same  → NOOP (skip)
hash_diff  → UPDATE (state_version++)
no_record  → CREATE (state_version = 1)
```

**spec_hash computation**: SHA-256 of canonical JSON of spec fields only:
```go
func SpecHash(spec interface{}) string {
    // 1. Marshal to JSON with sorted keys
    // 2. Remove non-spec fields (metadata, timestamps, IDs)
    // 3. SHA-256
}
```

**Fields included in hash** (per resource type):

| Resource | Hashed Fields |
|----------|---------------|
| Database | engine, version, instance_class, storage_gb, high_avail |
| Cache | engine, version, memory_mb, eviction_policy |
| ObjectStorage | acl, cors_rules |
| Compute | replicas, cpu, memory, port, env_vars |

**Fields excluded**: name (part of unique key), resource IDs, timestamps, status.

### ADR-5: infracfg.json generation strategy

**Decision**: Config Generator produces `infracfg.json` by mapping resource outputs to Encore's expected schema.

**Canonical field names** (以 PRD v1.1 §5.4 冻结字段为准，严格对齐 Encore 官方 `infracfg.rs` schema):

| PRD 冻结字段名 | JSON key | Go struct field | 类型 | P0 覆盖 |
|---------------|----------|----------------|------|---------|
| `sql_servers` | `sql_servers` | `SQLServers` | `map[string]SQLServer` | Yes |
| `redis` | `redis` | `Redis` | `map[string]RedisServer` | Yes |
| `object_storage` | `object_storage` | `ObjectStorage` | `map[string]ObjectStore` | Yes |

**注意**: Encore 的 `infracfg.rs` 使用 `map<resource_name, config>` 语义（key = Encore 代码中声明的资源名），不是数组。P0 实现必须使用 map 结构。

```go
// internal/infragen/generator.go
// 字段名严格对齐 PRD v1.1 §5.4 + Encore infracfg.rs
type InfraCfg struct {
    SQLServers    map[string]SQLServer    `json:"sql_servers,omitempty"`
    Redis         map[string]RedisServer  `json:"redis,omitempty"`
    ObjectStorage map[string]ObjectStore  `json:"object_storage,omitempty"`
}

// SQLServer 对齐 Encore infracfg.rs 中 SQLServer 结构
type SQLServer struct {
    Host     string `json:"host"`
    Port     int    `json:"port,omitempty"`
    DBName   string `json:"database"`
    User     string `json:"user"`
    Password string `json:"password"`  // injected via K8s Secret
    TLS      *TLS   `json:"tls,omitempty"`
}

// RedisServer 对齐 Encore infracfg.rs 中 RedisServer 结构
type RedisServer struct {
    Host     string `json:"host"`
    Port     int    `json:"port,omitempty"`
    Auth     string `json:"auth,omitempty"`
    KeyPrefix string `json:"key_prefix,omitempty"`
    TLS      *TLS   `json:"tls,omitempty"`
}

// ObjectStore 对齐 Encore infracfg.rs 中 ObjectStorage 结构
type ObjectStore struct {
    Provider  string `json:"provider"`  // "s3-compatible"
    Endpoint  string `json:"endpoint"`
    Bucket    string `json:"bucket"`
    Region    string `json:"region"`
    AccessKey string `json:"access_key"`
    SecretKey string `json:"secret_key"` // injected via K8s Secret
}
```

**映射示例** (Alicloud RDS → infracfg.json):
```json
{
  "sql_servers": {
    "mydb": {
      "host": "rm-bp1xxx.mysql.rds.aliyuncs.com",
      "port": 3306,
      "database": "mydb",
      "user": "infracast_mydb",
      "password": "${K8S_SECRET:mydb-password}"
    }
  }
}
```

**Injection chain**:
1. Config Generator writes `infracfg.json` (map 结构，key = 资源名)
2. CLI creates K8s ConfigMap from the file
3. K8s mounts ConfigMap into container at Encore's expected path
4. Encore Runtime reads `infracfg.json` at startup

**User override**: If `infra/infracfg.json` exists in project, CLI deep-merges it (user values win, per-resource-name granularity).

### ADR-6: Credential management via STS/AssumeRole

**Decision**: Primary credential flow uses STS temporary tokens.

```
┌──────────────┐    AssumeRole     ┌──────────────┐
│  Long-term   │──────────────────▶│    STS        │
│  AK/SK       │                   │  Service      │
│  (encrypted) │                   └──────┬───────┘
└──────────────┘                          │
                                          │ Temporary credentials
                                          │ (15min - 1hr TTL)
                                          ▼
                               ┌──────────────────┐
                               │  SDK Direct calls │
                               │  (RDS/OSS/Redis/  │
                               │   ACK/ACR)        │
                               └──────────────────┘
```

**Storage**:
- Long-term AK/SK: AES-256-GCM encrypted in state DB
- Temporary tokens: in-memory only, never persisted
- Key derivation: PBKDF2 from user-provided passphrase (local mode) or KMS (team mode)

**Fallback**: Direct AK/SK for development/testing (no STS call). Controlled by `--credentials-mode` flag.

## 7. Error Handling and Rollback Strategy

### 7.1 Pipeline Error Model

```go
// internal/deploy/errors.go
type PipelineError struct {
    Step      PipelineStep  // build | map | provision | configure | deploy | verify
    Resource  string        // which resource failed (if applicable)
    Cause     error
    Retryable bool
}

type PipelineStep int
const (
    StepBuild PipelineStep = iota
    StepMap
    StepProvision
    StepConfigure
    StepDeploy
    StepVerify
    StepNotify
)
```

### 7.2 Failure Recovery Matrix

| Step | Failure | Recovery | Data Impact |
|------|---------|----------|-------------|
| **Build** | `encore build` fails | Abort. User fixes code. | None |
| **Map** | Invalid config | Abort. User fixes `infracast.yaml`. | None |
| **Provision** | SDK call fails | Retry up to 3x with exponential backoff. Partial resources remain (state tracked). | Resources may be partially created. Next `deploy` resumes from state. |
| **Configure** | Schema mismatch | Abort. Log expected vs actual fields. | Provisioned resources remain (usable on retry). |
| **Deploy** | ACR push fails | Retry. | No K8s change yet. |
| **Deploy** | K8s apply fails | Retry. If persistent → abort, resources remain. | No traffic shift. |
| **Verify** | Health check timeout (5min) | Rollback K8s Deployment to previous revision. | DB state preserved (forward-only). App reverted. |
| **Notify** | Webhook fails | Log warning. Non-blocking. | Deploy succeeded regardless. |

### 7.3 Forward-Only Migration (Constraint C7)

```
Deploy v2 with migration_003.sql
    │
    ├── migration succeeds → continue deploy
    │
    └── migration fails
         ├── DB state: migration_003 partially applied → manual fix required
         ├── App state: rollback to v1 Deployment (which works with migration_002 schema)
         └── Next deploy: fix migration_003 → push v3
```

**No destructive rollback**: We never run `DROP` / `ALTER TABLE ... DROP COLUMN` as rollback. The DB only moves forward. App version can roll back because Encore apps should be backward-compatible with the previous schema version.

## 9. Security Architecture

### 9.1 Threat Model (P0 scope)

| Threat | Mitigation |
|--------|-----------|
| Credential leakage (AK/SK in logs/files) | AES-256-GCM encryption at rest; STS temporary tokens in memory only; structured logging with credential redaction |
| Unauthorized cloud access | STS AssumeRole with scoped IAM policy; minimum privilege per resource type |
| State file tampering | SQLite file permissions 0600; team mode uses PostgreSQL with auth |
| Supply chain attack (compromised dependencies) | `go.sum` integrity; CI runs `go mod verify`; image signing in P1 |
| Insecure K8s deployment | K8s Secrets for sensitive config; NetworkPolicy for service isolation |

### 9.2 Credential Flow

```go
// internal/credentials/manager.go
type CredentialManager interface {

type Credentials struct {

    AccessKeySecret string

    Expiration      time.Time // when STS token expires
}

        Config:    testConfig,

    })
    assert.NoError(t, err)
    assert.Equal(t, "live", result.Status)
}
```

### 10.3 Integration Test Strategy (Milestone B+)

- **Alicloud integration tests**: Run against real Alicloud APIs with a dedicated test account
- **Resource cleanup**: Each test creates resources with prefix `infracast-test-{timestamp}` and destroys them in `t.Cleanup()`
- **CI gate**: Integration tests run on dedicated CI job (not on every PR, only on merge to main)

## 14. Risk Gates (Architecture-level)

| Gate | Verification Method | Fail Action |
|------|-------------------|-------------|
| `infracfg.json` compatible with Encore Runtime | Integration test: generate config → start Encore app → verify it reads config | Stop. Return to PoC. |
| ACK Serverless deploy works end-to-end | E2E test: push image → deploy → health check passes | Stop. Verify K8s manifest. |
| spec_hash idempotency correct | Unit test: same spec → same hash; different spec → different hash; metadata changes → same hash | Stop. Fix hash computation. |
| STS temporary credentials work with all resource types | Integration test: STS token → create RDS + OSS + Redis + ACK deploy | Stop. Check IAM policy. |
| State Store handles concurrent access | Unit test: two goroutines updating same resource → state_version conflict detected | Stop. Fix optimistic locking. |
